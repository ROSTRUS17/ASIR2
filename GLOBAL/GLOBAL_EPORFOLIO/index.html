ATTACK_SIMULATION_RA123

  1. Summary: We used **Kali Purple** to simulate brute-force attacks. First, we generated custom password dictionaries using **Pydictor** (creating number and letter lists) and merged them with **Dymerge**. Then, we used these dictionaries with **Hydra** to attack a local SSH server and a web login form (DVWA).
  2. Complex Concepts Learned:
    • Dictionary Attack workflow: Understanding that a brute-force attack is only as good as the wordlist you use. Merging lists is key.
    • Hydra Syntax: Learning the specific flags for Hydra (like `-L` for user list vs `-l` for single user, and the http-post-form syntax) was tricky.
  3. How I Felt: **SUPER COOL üòé**. Watching Hydra crack the password and show the green text was very satisfying. I felt like Mr. Robot, although configuring the SSH server at the beginning was a bit boring.
  4. What I'd Improve: My dictionary was too small/simple. In a real scenario, I should use bigger wordlists like *rockyou.txt* or generate more complex permutations with Pydictor.
  5. Glossary words: *Brute Force*, *Wordlist*, *SSH (Secure Shell)*, *HTTP POST*, *Base64*.
  6. Challenge Statement Review:
    • Grade: 9/10.
    • Why: Very practical.
    • Improvement: The instructions were vague.

COMAN2_LINUX_RA1

  1. Summary: A massive exercise on managing users, groups, and permissions. We worked with `chmod`, `chown`, and special permissions like **Sticky Bit**, **SUID**, and **SGID** in collaborative folders. We also practiced finding files using `grep` (content), `find` (properties), and `locate` inside a simulated directory structure with logs and scripts.
  2. Complex Concepts Learned:
    • ACLs (Access Control Lists): Standard permissions (rwx) are not enough when you need a specific user to read a file but not the rest of the group. `setfacl` saved my life here.
    • Sticky Bit vs SGID: SGID inherits the group (good for shared folders), but Sticky Bit prevents users from deleting each other's files (good for /tmp). Confusing at first!
  3. How I Felt: **OVERWHELMED**. There were so many commands. The `grep` regex part (`^#`) to find lines without comments took me a while to figure out.
  4. What I'd Improve: I should memorize the numeric codes for permissions (like 777, 644, 1777) better so I don't have to calculate them manually every time.
  5. Glossary words: *ACL*, *Sticky Bit*, *SGID*, *Recursive Search*, *Regex*, *Umask*.
  6. Challenge Statement Review:
    • Grade: 10/10.
    • Why: It covers absolutely everything about Linux administration.
    • Improvement: Maybe split it into two parts, it was very long for a single session.

MALWARES_RA123

  1. Summary: We had to research and present in a team about different malware types: **Cryptojacking**, **Adware**, **Ransomware**, etc. The rule was "no reading from slides". We had to explain the definition, attack techniques, and real cases.
  2. Complex Concepts Learned:
    • Cryptojacking: How hackers use your CPU to mine cryptocurrency without you knowing. I didn't know this could happen via web scripts (Coinhive).
    • Social Engineering in Adware: How adware isn't just annoying pop-ups but often tricks users into installing worse malware.
  3. How I Felt: üò® **NERVOUS**. I hate public speaking. But researching the real cases was actually interesting.
  4. What I'd Improve: Our slides had too much text. Next time, more images and diagrams, and less bullet points so we are forced not to read.
  5. Glossary words: *Cryptojacking*, *Ransomware*, *Adware*, *Backdoor*, *Social Engineering*.
  6. Challenge Statement Review:
    • Grade: 7/10.
    • Why: Good for soft skills, but I prefer coding.
    • Improvement: Allow a recorded video presentation instead of live speaking.

EXPLOIT_BOOM_RA123

  1. Summary: We developed two vulnerable web apps (one in Python/Flask, one in PHP). We performed **SQL Injection** attacks to bypass login screens using payloads like `' OR 1=1 --`. Then we did **XSS (Cross-Site Scripting)** attacks. Finally, we fixed the code using "Prepared Statements" and input sanitization (`htmlspecialchars`).
  2. Complex Concepts Learned:
    • Prepared Statements: Why concatenating strings in SQL is a deadly sin. Parameterized queries separate code from data.
    • Reflected vs Persistent XSS: The difference between a script that executes once (search bar) vs one that stays in the database and hacks everyone who visits the page.
  3. How I Felt: üòà **POWERFUL**. Bypassing a login without a password felt magical. Fixing the code made me feel like a responsible developer.
  4. What I'd Improve: My PHP code for the fix was a bit messy. I should organize the files better instead of putting logic and HTML in the same file.
  5. Glossary words: *SQL Injection*, *XSS (Cross-Site Scripting)*, *Payload*, *Sanitization*, *Prepared Statements*.
  6. Challenge Statement Review:
    • Grade: 5/10.
    • Why: Wasn' in the mood.
    • Improvement: Too complicated.

üíæ DATABASE ADMINISTRATION


DBA_MYSQL_RA12: Server Configuration

  1. Summary: I acted as a DBA consultant for 3 different companies. I had to configure 15 specific MySQL parameters (like `innodb_buffer_pool_size`, `max_connections`, `slow_query_log`) based on interview transcripts. I explained the "why" for each value in a table and modified the configuration files.
  2. Complex Concepts Learned:
    • Buffer Pool Size: Calculating the correct RAM allocation (usually 70-80% of dedicated RAM) so the server doesn't crash.
    • Slow Query Log: How to enable logging for queries that take longer than `long_query_time` to execute, which is essential for performance tuning.
  3. How I Felt: üòµ **CONFUSED** at first with so many parameters. I didn't know where the config file was (`/etc/mysql/my.cnf` vs `/etc/mysql/mysql.conf.d/`). But satisfying when the service restarted correctly.
  4. What I'd Improve: I would automate the calculation of memory parameters using a script instead of doing it manually with a calculator.
  5. Glossary words: *InnoDB Buffer Pool*, *My.cnf*, *Slow Query Log*, *Bind Address*, *Performance Schema*.
  6. Challenge Statement Review:
    • Grade: 8/10.
    • Why: Very realistic scenario (consulting).
    • Improvement: Include more specific RAM limits for the fictional companies to make the math harder.

USERS_MYSQL_RA23: User Management & "Red Social"

  1. Summary: We created a "Social Network" database with 3 tables (User, Group, Comment) and FK relationships. Then we managed 3 users (Bego, Mati, Mifli) granting them very specific permissions (SELECT, UPDATE, REVOKE). We also tested access from different domains and changed passwords.
  2. Complex Concepts Learned:
    • FLUSH PRIVILEGES: Why it's necessary when you modify the grant tables directly but not when you use `GRANT/REVOKE` commands.
    • GRANT OPTION: The ability to let a user (like 'Crispula') give permissions to *other* users. It's dangerous if not controlled!
  3. How I Felt: üëÆ **LIKE A POLICE OFFICER**. Controlling exactly who can delete data and who can only read was fun. I hated the syntax errors when typing `IDENTIFIED BY`.
  4. What I'd Improve: I repeated the same commands many times. I should have created a `.sql` script to run everything at once.
  5. Glossary words: *DCL (Data Control Language)*, *Grant Option*, *Flush Privileges*, *Foreign Key (FK)*, *Revoke*.
  6. Challenge Statement Review:
    • Grade: 9/10.
    • Why: Very complete, covers all CRUD permissions.
    • Improvement: The part about creating the specific tables was a bit repetitive to type manually.

DBA_SQLITE_RA12: "Frutitas Bonitas" Optimization

  1. Summary: We managed a local SQLite database for a fruit app (Mandarinas, Melocotones, Caquis). We had to create tables, analyze storage size, optimize performance using logs and `EXPLAIN QUERY PLAN`, and set up automated backups.
  2. Complex Concepts Learned:
    • PRAGMA: Special commands in SQLite to change how the DB behaves (like `PRAGMA synchronous` or `PRAGMA journal_mode`).
    • WAL (Write-Ahead Logging): A journaling mode that allows simultaneous reading and writing, which is much faster than the default DELETE mode.
  3. How I Felt: üçä **AMUSED** by the fruit names, but surprised by how complex SQLite can be. I thought it was just a simple file.
  4. What I'd Improve: I didn't fully understand the "explain query plan" output at first. I need to study more about database indexing.
  5. Glossary words: *Pragma*, *WAL (Write-Ahead Log)*, *Explain Query Plan*, *Auto-increment*, *Journaling*.
  6. Challenge Statement Review:
    • Grade: 7/10.
    • Why: Good concepts, but SQLite feels less "pro" than MySQL.
    • Improvement: Provide a larger dataset (thousands of fruits) to really see the performance difference.

POSTGRES_DBA_RA34: Equitation DB & Advanced Features

  1. Summary: We installed PostgreSQL on Linux, created a Horse Riding database ("Equitacion Suave"), and managed Roles using both CLI and **pgAdmin**. We also implemented advanced features like JSON columns, Arrays, and Materialized Views.
  2. Complex Concepts Learned:
    • JSONB in SQL: Storing JSON data inside a table column. It allows NoSQL flexibility within a relational database.
    • Materialized Views: Unlike normal views, these store the data physically to make complex queries faster, but you have to refresh them manually.
  3. How I Felt: ü§© **IMPRESSED**. pgAdmin is a very nice tool, much better than the command line. Using Arrays in a table felt weird coming from MySQL.
  4. What I'd Improve: The inheritance part (table inheritance) was confusing. I'm not sure when I would use that in real life instead of a Foreign Key.
  5. Glossary words: *JSONB*, *Materialized View*, *pgAdmin*, *Role*, *Table Inheritance*.
  6. Challenge Statement Review:
    • Grade: 10/10.
    • Why: It covered everything from installation to very advanced topics.
    • Improvement: None, it was excellent.

DA2_RA15: Dice Game App

  1. Summary: I built a web application using **PHP** and HTML to simulate a dice game between two players. It has two pages: a **form** to select the number of dice, faces, and opponent points, and a **results page** that generates random numbers, sums them up, and declares a winner.

EXTENSIONES_VSC_PYTHON_RA1: VS Code & Python Setup

  1. Summary: I conducted a study on the best **VS Code extensions** for Python development. I selected the top 3 extensions (likely the core *Python* extension, Pylance, and a formatter like Black) and justified why I kept or discarded several others. The result was a **Markdown (.md)** document with an index in my Git repository.
  2. Complex Concepts Learned:
    • **Language Server Protocol (LSP):** Understanding that extensions like Pylance use the LSP to provide powerful features like code completion and error checking without relying only on the Python interpreter.
    • **Virtual Environments:** Realizing the importance of selecting the correct Python interpreter (often inside a virtual environment) within VS Code for everything to work correctly.
  3. How I Felt: ü§ì **ORGANIZED**. This task forces you to set up your environment correctly from the start. I found Pylance much better than the old language server.
  4. What I'd Improve: The process of justifying why I discarded certain extensions felt a bit arbitrary if they were not bad, just less popular.
  5. Glossary words: *Pylance*, *Virtual Environment*, *LSP (Language Server Protocol)*, *Interpreter*, *Formatter*.
  6. Challenge Statement Review:
    • Grade: 8/10.
    • Why: Essential step for Python development.
    • Improvement: Should integrate with the Git extension study for a complete workflow.

NAVARREDONDA_PYTHON_RA125: Amusement Park Ticket Simulator

  1. Summary: I developed a Python program to calculate the final price of an amusement park ticket based on 7 input parameters and 12 prioritized modification rules (from free entry for seniors/members to fixed weekend surcharges). The core challenge was implementing the **priority system** correctly, where only the highest priority rule that is met applies.
  2. Complex Concepts Learned:
    • **Conditional Logic (`if`/`elif`/`else`):** Mastering the use of `elif` (else if) to ensure that once a high-priority rule is met (e.g., Free Entry, Rule 1), the program stops checking the lower-priority rules (e.g., 50% discount, Rule 2).
    • **Random Simulation:** Implementing the automatic input for "Day of the week" and "Is a club member" using the `random` module to test different scenarios.
  3. How I Felt: üß† **FOCUSED**. This was a pure logic puzzle. It was tricky handling the floating-point math (decimals) correctly and making sure the output format was exactly as requested.
  4. What I'd Improve: The price calculation logic became a huge block of `if/elif`. I could probably refactor it into separate functions to make it cleaner.
  5. Glossary words: *elif*, *Floating Point Arithmetic*, *Random Module*, *Input Validation*, *Prioritized Logic*.
  6. Challenge Statement Review:
    • Grade: 10/10.
    • Why: Excellent practice for real-world business logic and complex priority rules.
    • Improvement: None, it was a solid programming challenge.

FASTEST_TURTLE_RA12345: Creative Graphics with Turtle

  1. Summary: I used the Python **Turtle** graphics library to complete three exercises: 1) Draw the La Salle star, 2) Write my name without using the `write()` function (by drawing letters with lines), and 3) Create a complex, creative drawing using over 30 function calls (many via `for` or `while` loops).
  2. Complex Concepts Learned:
    • **Procedural Graphics:** Understanding how simple commands (`forward`, `left`, `right`) can create complex shapes when combined with iterative loops (`for` loops and modular arithmetic: `i % 90`).
    • **State Management:** Using functions like `penup()` and `pendown()` to move the drawing cursor without leaving a trace, which was essential for "writing" the name.
  3. How I Felt: üé® **CREATIVE**. It was a fun break from pure logic. I enjoyed making the complex pattern in Exercise 3 (the one that uses `forward(i%90)`). It was visually stunning.
  4. What I'd Improve: The installation process for `tkinter` can sometimes be tricky on different Linux distributions if it's not pre-installed.
  5. Glossary words: *Turtle*, *Tkinter*, *Procedural Generation*, *Modular Arithmetic*, *Pen Up/Down*.
  6. Challenge Statement Review:
    • Grade: 9/10.
    • Why: Great way to learn basic Python structure (loops, functions) in a visual way.
    • Improvement: Include an exercise that requires user interaction (e.g., drawing based on keyboard input).
